# Plugin Seam (unplugin) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the bundler-specific `@devgent/vite-plugin` glue with a bundler-agnostic [unplugin](https://github.com/unjs/unplugin) factory (`@devgent/plugin-core`) plus one thin re-export package per bundler (`@devgent/plugin-{vite,webpack,rspack,rollup,esbuild}`). Each bundler entry only boots the standalone `@devgent/core` engine once on dev-server start and injects the widget's `<head>` tags; per-bundler code is reduced to the HTML-injection mechanism. Migrate the example app and keep the Playwright browser IT green.

**Architecture:** `@devgent/plugin-core` calls `createUnplugin((opts: DevgentOptions) => ({ name: 'devgent', vite, webpack, rspack, rollup, esbuild }))`. On dev-server start (per-bundler hook) it boots `@devgent/core` `start(opts) => {port, stop}` exactly once, stores `{port, stop}`, and stops on server close; it injects `core.htmlTags(corePort, opts)` head tags (meta `pw-api-base` = core port, meta `pw-preview-id`, deferred widget `<script>`). Vite injects via `transformIndexHtml(order:'pre')`; webpack/rspack tap the `HtmlWebpackPlugin`/`HtmlRspackPlugin` compilation hook; rollup/esbuild are documented build-only no-ops. A `devgent` PATH-shim helper (moved out of the old `plugin.ts`) lets the spawned agent resolve `devgent tools`. The five per-bundler packages each contain ONE named file re-exporting the matching `unplugin.<bundler>` entry.

**Tech Stack:** ESM-only, Node 22, TypeScript with `verbatimModuleSyntax` + `isolatedModules` + `noUncheckedIndexedAccess`. Functions not classes; no IIFEs; **no `index.ts` barrels** (named files only, package `exports` subpaths point at named files). Build = `tsdown`, test = `vitest run`, typecheck = `tsc --noEmit`, lint = `oxlint`. Monorepo via pnpm workspaces + turbo. Dependency: `unplugin`.

## Assumes Plans 1–3 complete

This plan **assumes Plans 1–3 are complete**, i.e.:

- `@devgent/core` exists and exposes:
  - `start(opts: DevgentOptions) => Promise<{ port: number; stop: () => Promise<void> }>` — boots the standalone h3 engine (all `/__pw/*` routes, lock, session, uiBus, harness + runner registries wired) and returns the listening port plus a stop handle.
  - `htmlTags(corePort: number, opts: DevgentOptions) => HtmlTag[]` — returns the framework-neutral head tags: `<meta name="pw-api-base" content="http://localhost:${corePort}">`, `<meta name="pw-preview-id" content="${previewId}">`, and (when a widget URL resolves) `<script src="${widgetUrl}" defer>`. `HtmlTag` is `{ tag: string; attrs: Record<string, string | boolean>; injectTo: 'head' }`.
  - `resolveWidgetUrl(opts) => string | undefined` — resolves the bundled `@devgent/widget/global` URL the core engine serves, or `opts.widgetUrl`, or `undefined`. (Core serves the widget itself now that the engine is standalone; `htmlTags` uses this internally.)
- Harness and test-runner adapters resolve via registries inside `@devgent/core`; the plugin passes `opts.harness` / `opts.testRunner` through untouched.
- The old per-route glue (`chat-route.ts`, `tools-route.ts`, `vitest-*`, `ui-bus.ts`, `widget-middleware.ts`, etc.) has moved into `@devgent/core`. **This plan only replaces the bundler-specific plugin layer.**

If `@devgent/core` does not yet export `start` / `htmlTags` / `resolveWidgetUrl` with these signatures, STOP and complete Plans 1–3 first.

- [ ] **Minimal impl** — write `packages/plugin-core/src/bin-shim.ts` (ports lines 91–104 of the old `plugin.ts`):

  ```ts
  import {mkdirSync, rmSync, symlinkSync} from 'node:fs'
  import {join, delimiter} from 'node:path'
  import {createRequire} from 'node:module'

  const require = createRequire(import.meta.url)

  // The agent calls `devgent tools …` from Bash, but `devgent` isn't on PATH. Drop a shim on
  // the spawned agent's PATH → the @devgent/cli bin, so `devgent tools` resolves. Returns the
  // bin dir to prepend to PATH (callers pass it to core.start as the harness PATH prefix). Best
  // effort: without the shim (e.g. @devgent/cli not installed) the agent falls back to whatever
  // `devgent` is already on PATH.
  export function installDevgentBinShim(stateDir: string): string {
    const binDir = join(stateDir, 'bin')
    mkdirSync(binDir, {recursive: true})
    const shim = join(binDir, 'devgent')
    try {
      const binPath = require.resolve('@devgent/cli/bin')
      rmSync(shim, {force: true})
      symlinkSync(binPath, shim)
    } catch {
      // best effort — see above.
    }
    return `${binDir}${delimiter}${process.env.PATH ?? ''}`
  }
  ```

- [ ] Write `packages/plugin-core/src/plugin.ts` — the `createUnplugin` factory. The Vite hook ports the old boot logic: boot `core.start()` once (memoized per plugin instance), stop on server close, inject `core.htmlTags()` via `transformIndexHtml(order:'pre')`. Other bundler hooks are stubbed here as TODO placeholders filled by Tasks 3–4 (so this file compiles now):

  ```ts
  import {join} from 'node:path'
  import {createUnplugin, type UnpluginFactory} from 'unplugin'
  import {start, htmlTags} from '@devgent/core'
  import {installDevgentBinShim} from './bin-shim.js'
  import type {DevgentOptions} from './options.js'

  // A booted engine handle, memoized per plugin instance so repeated dev-server hooks (vite's
  // transformIndexHtml runs per html response) boot @devgent/core exactly once.
  interface Engine {
    port: number
    stop: () => Promise<void>
  }

  // Boot @devgent/core once for this plugin instance. The PATH shim is installed into the
  // engine's state dir and handed to core via agentPathPrefix so the spawned harness resolves
  // `devgent tools`. Subsequent callers await the same in-flight/booted promise.
  function makeEngineBooter(opts: DevgentOptions, root: string): () => Promise<Engine> {
    let booting: Promise<Engine> | null = null
    return () => {
      if (booting) return booting
      const lockDir = opts.lockDir ?? root
      const stateDir = join(lockDir, '.devgent')
      const agentPathPrefix = installDevgentBinShim(stateDir)
      booting = start({...opts, lockDir, root, agentPathPrefix}).then((handle) => ({
        port: handle.port,
        stop: handle.stop,
      }))
      return booting
    }
  }

  export const unpluginFactory: UnpluginFactory<DevgentOptions | undefined> = (rawOptions) => {
    const opts: DevgentOptions = rawOptions ?? {}
    let booter: (() => Promise<Engine>) | null = null
    let engine: Engine | null = null

    return {
      name: 'devgent',
      // ----- Vite -----
      vite: {
        // Dev only; a no-op for production builds.
        apply: 'serve',
        // order:'pre' so our tags land before framework html middleware writes the document,
        // matching the old plugin. The handler boots the engine on first html transform and
        // reuses the booted port thereafter.
        transformIndexHtml: {
          order: 'pre',
          async handler(_html: string, ctx: {server?: {config: {root: string}}}) {
            if (opts.enabled === false) return []
            const root = ctx.server?.config.root ?? process.cwd()
            booter ??= makeEngineBooter(opts, root)
            engine ??= await booter()
            return htmlTags(engine.port, opts)
          },
        },
        configureServer(server: {
          httpServer?: {on(event: 'close', cb: () => void): void} | null
          config: {root: string}
        }) {
          if (opts.enabled === false) return
          booter ??= makeEngineBooter(opts, server.config.root)
          // Boot eagerly so SSR frameworks with no static index.html still get a live engine,
          // and register teardown. transformIndexHtml reuses the same memoized handle.
          void booter().then((e) => {
            engine = e
          })
          server.httpServer?.on('close', () => {
            void engine?.stop()
          })
        },
      },
      // ----- webpack ----- (Task 3 fills this in)
      // ----- rspack ----- (Task 3)
      // ----- rollup ----- (Task 4)
      // ----- esbuild ----- (Task 4)
    }
  }

  export const unplugin = /* #__PURE__ */ createUnplugin(unpluginFactory)
  ```

  > The `engine`/`booter` closures are shared across all bundler hooks of one plugin instance, so whichever hook fires first boots the engine and the rest reuse it. The shared HTML-injection helper added in Task 3 reads `engine.port` + `opts`.

  Add this helper above `unpluginFactory` (renders the engine's `htmlTags` into the HTML string the head-tag injection plugins consume). The compiler is described by **structural interfaces** and rspack's HTML plugin is reached through a **type guard**, so no `as` cast is needed:

  ```ts
  // Render core.htmlTags into HtmlWebpackPlugin/HtmlRspackPlugin "tag" descriptors. Both html
  // plugins accept the same {tagName, attributes, voidTag, meta} shape on alterAssetTagGroups.
  interface HtmlTag {
    tag: string
    attrs: Record<string, string | boolean>
    injectTo: 'head'
  }
  interface AssetTag {
    tagName: string
    voidTag: boolean
    attributes: Record<string, string | boolean>
    meta: {plugin: string}
  }
  function toAssetTags(tags: HtmlTag[]): AssetTag[] {
    return tags.map((t) => ({
      tagName: t.tag,
      voidTag: t.tag === 'meta',
      attributes: t.attrs,
      meta: {plugin: 'devgent'},
    }))
  }

  // The HtmlWebpackPlugin/HtmlRspackPlugin head-tag hook surface, shared by both bundlers.
  type AlterAssetTagGroups = {
    tapPromise(name: string, cb: (data: {headTags: AssetTag[]}) => Promise<{headTags: AssetTag[]}>): void
  }
  type HtmlTagHooks = {alterAssetTagGroups: AlterAssetTagGroups}
  // The compilation object the webpack/rspack `getHooks(compilation)` call expects. We pin the
  // tap callback's parameter to exactly what HtmlWebpackPlugin.getHooks accepts (no cast).
  type Compilation = Parameters<typeof HtmlWebpackPlugin.getHooks>[0]

  // Structural compiler shape: just the bits both bundlers expose. `context` is the project root;
  // `webpack` is rspack's webpack-compat namespace that carries HtmlRspackPlugin.
  interface BundlerCompiler {
    context?: string
    hooks: {thisCompilation: {tap(name: string, cb: (compilation: Compilation) => void): void}}
    webpack?: {HtmlRspackPlugin?: {getHooks(c: Compilation): HtmlTagHooks}}
  }
  // Type guard: does this rspack-compat namespace actually carry HtmlRspackPlugin.getHooks?
  function hasRspackHtmlPlugin(
    ns: BundlerCompiler['webpack'],
  ): ns is {HtmlRspackPlugin: {getHooks(c: Compilation): HtmlTagHooks}} {
    return typeof ns?.HtmlRspackPlugin?.getHooks === 'function'
  }

  // Shared push: boot the engine once, then append the widget's head tags. Reused by both hooks.
  async function pushDevgentTags(
    data: {headTags: AssetTag[]},
    bootEngine: () => Promise<Engine>,
    opts: DevgentOptions,
  ): Promise<{headTags: AssetTag[]}> {
    const engine = await bootEngine()
    data.headTags.push(...toAssetTags(htmlTags(engine.port, opts)))
    return data
  }
  ```

  Now add the `webpack` and `rspack` hooks to the returned object (after the `vite` block). The factory's `engine`/`booter` closures are reused. `webpack(compiler)` and `rspack(compiler)` receive the compiler:

  ```ts
      // ----- webpack -----
      // `compiler` is typed structurally (BundlerCompiler), so `.context` and `.hooks` are read
      // without a cast. The thisCompilation tap callback's `compilation` is pinned to exactly
      // HtmlWebpackPlugin.getHooks's parameter type, so getHooks(compilation) needs no cast.
      webpack(compiler: BundlerCompiler) {
        if (opts.enabled === false) return
        const root = compiler.context ?? process.cwd()
        booter ??= makeEngineBooter(opts, root)
        const ensure = booter
        compiler.hooks.thisCompilation.tap('devgent', (compilation) => {
          const hooks = HtmlWebpackPlugin.getHooks(compilation)
          // tapPromise: boot the engine (once) before tags are emitted, then push our tags.
          hooks.alterAssetTagGroups.tapPromise('devgent', (data) =>
            pushDevgentTags(data, () => (engine ? Promise.resolve(engine) : ensure().then((e) => (engine = e))), opts),
          )
        })
      },
      // ----- rspack -----
      rspack(compiler: BundlerCompiler) {
        if (opts.enabled === false) return
        const root = compiler.context ?? process.cwd()
        booter ??= makeEngineBooter(opts, root)
        const ensure = booter
        compiler.hooks.thisCompilation.tap('devgent', (compilation) => {
          // @rspack/core ships HtmlRspackPlugin with the SAME getHooks(compilation) →
          // alterAssetTagGroups API surface as html-webpack-plugin. Resolve it lazily off the
          // host's installed @rspack/core (the `webpack` compat namespace) so plugin-core doesn't
          // hard-depend on rspack. A type guard narrows the namespace — no cast.
          if (!hasRspackHtmlPlugin(compiler.webpack)) return
          const hooks = compiler.webpack.HtmlRspackPlugin.getHooks(compilation)
          hooks.alterAssetTagGroups.tapPromise('devgent', (data) =>
            pushDevgentTags(data, () => (engine ? Promise.resolve(engine) : ensure().then((e) => (engine = e))), opts),
          )
        })
      },
  ```

  > rspack exposes `HtmlRspackPlugin` on `compiler.webpack` (its webpack-compat namespace), so we resolve it off the compiler via the `hasRspackHtmlPlugin` guard rather than importing `@rspack/core` — keeping it a peer concern of `@devgent/plugin-rspack`. Both hooks share the same `pushDevgentTags` path (boot-once + `toAssetTags` + `htmlTags`); the only difference is which `getHooks` they call. No `as` casts: the compiler is typed structurally and the rspack html plugin is reached through a guard.
